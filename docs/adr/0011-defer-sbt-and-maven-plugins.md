# ADR-0011 — Defer sbt and Maven plugin siblings to post-v0.1

**Status:** Accepted

## Context

ADR-0004 commits the architecture to a build-tool-agnostic `deps-lib` so sbt, Maven, and standalone-CLI wrappers can sit next to the Gradle plugin without duplicating resolver logic. The original plan (M6) listed "sbt plugin" and "Maven plugin" as siblings to the existing `gradle-plugin`.

During the M6 push we reached the point where the integration surface was fully ready — `ManifestProducer.produceSimple(localRepoDir, lang, coords, exclusions...)` in `deps-lib/src/full/` does the transitive resolution and SHA-pinning; `ManifestIo.write` emits the TOML — but the two plugins themselves were not implemented. Shipping them responsibly needs:

- **sbt plugin:** a Scala-built subproject depending on `sbt-plugin-api` (which pins Scala 2.12 for sbt 1.x), an sbt scripted test harness, and publishing to the sbt Plugin Index. The whole repo is otherwise Java 21, so this is the first Scala 2.12 subproject — non-trivial wiring.
- **Maven plugin:** a Java subproject depending on `maven-plugin-api` + `maven-plugin-annotations` with the `plugin.xml` descriptor generated by an annotation processor, plus a `maven-invoker-plugin` integration-test harness, plus publishing via the `org.apache.maven.plugins:maven-plugin-plugin` packaging type. The Gradle `maven-publish` plugin does not produce a valid Maven plugin artifact on its own — custom packaging is required.

Neither plugin changes the runtime semantics of mc-lib-provider. They exist to let mod authors whose build isn't Gradle participate. For the foreseeable near-term adoption (Scala + Kotlin mods for NeoForge/Fabric 1.21.1+), the overwhelming majority of mod projects use Gradle — Loom and ModDevGradle are Gradle-only, and the mods we're targeting need one of those on the critical path regardless.

## Decision

v0.1 ships with the Gradle plugin only. The sbt and Maven plugins move to a post-v0.1 backlog item, unblocked whenever a real mod author asks for one or a contributor picks them up.

Both are greenfield subprojects; no changes to `deps-lib`, `core`, or the Gradle plugin are needed to enable them later. The hand-off is exactly the two `ManifestProducer` static/instance methods:

```java
// One-shot convenience — mirrors the 90% case used by the Gradle plugin.
Manifest m = ManifestProducer.produceSimple(
    localRepoDir,             // build-tool-scoped cache dir
    "scala",                  // lang
    coords,                   // List<String> of groupId:artifactId:version
    "net.minecraft:*",        // exclusions (varargs)
    "net.fabricmc:*",
    "com.mojang:*");
ManifestIo.write(m, outputFile);
```

For sbt, the entire task body is ~30 lines of Scala that reads `libraryDependencies.value`, maps to coord strings, and invokes the two calls above.

For Maven, a `@Mojo(name = "generate-mclib-manifest", defaultPhase = LifecyclePhase.PROCESS_RESOURCES)` Mojo reads `MavenProject#getDependencyArtifacts` and does the same. Roughly 60 lines of Java including imports + constructors.

## Consequences

**Positive:**
- v0.1 ships focused — one build-tool integration that's been exercised end-to-end, rather than three partially-validated ones.
- No new build infrastructure to maintain. Adding sbt or Maven to the root Gradle build would each pull in a new toolchain (Scala 2.12 for sbt, Maven lifecycle for the Maven plugin's scripted tests) that has to be maintained even when nobody uses it.
- The `deps-lib` / `ManifestProducer` contract stays the load-bearing interface. When someone implements either sibling, the contract won't churn.

**Negative:**
- Non-Gradle mod authors can't adopt mc-lib-provider at v0.1 without writing their own glue or invoking `ManifestProducer` by hand. Mitigation: document the direct-API usage in the README.
- "We support Scala/Kotlin/Java mods on Fabric + NeoForge" is slightly weakened by "…as long as you're on Gradle." Reality check: Fabric Loom and ModDevGradle *only* exist for Gradle, so anyone targeting Fabric or NeoForge is already on Gradle by construction. The hypothetical sbt-building Scala mod author would be using their own reinvented Loom port regardless.

## Re-opening

Reopen this ADR (status → Superseded) when any of these is true:

1. A mod author asks for non-Gradle support on the issue tracker.
2. Someone starts a port of Loom or ModDevGradle to sbt/Maven (unlikely, but would bring sbt-side mod authors into scope).
3. We want `deps-lib` to be useful as a CLI tool outside the Minecraft context — in which case a generic Maven plugin becomes the natural shape.

Until then: document, wait, revisit.
