# ADR-0019: Bridge manifest format and runtime registration

**Status:** Accepted — supersedes the runtime-wiring and manifest-discovery errata in ADR-0018.

## Context

ADR-0018 specifies the auto-codegen mixin-bridge pipeline: the Gradle plugin scans compiled mixin bytecode, rewrites cross-classloader call sites to dispatch through bridge interfaces, and emits matching bridge interface + impl classes. To wire those at runtime, each rewritten mixin gains a synthetic `<clinit>` that populates per-target `LOGIC_*` static fields by calling back into `McdpProvider.resolveAutoBridgeImpl(mixinFqn, fieldName)`. The provider needs a registry mapping `(mixinFqn, fieldName)` to the bridge impl FQN + the per-mod `ModClassLoader` to load it through.

How that registry is populated has been revisited three times before this ADR:

1. **`registerMod` URL scan** (initial in 0018). `McdpProvider.registerMod(modId, loader)` walked `loader.getURLs()` looking for `META-INF/mcdp-mixin-bridges/<mixinFqn>.txt` resources, parsing each as `bridge=…\nimpl=…\nfield=…\n` triplets. Worked for production jars where the mod jar URL was on the per-mod classloader.
2. **`registerAutoBridgeManifests(modLoader, manifestDir)` directory scan** (post-v0.1.0 errata). NeoForge dev exposed a hole: `IModFile.getFilePath()` returns one path per mod, but FML's mod-content view is multi-rooted in dev (the bridge-manifest dir typically lives under `build/resources/main`, separate from the classes dir on the per-mod loader). Adapters started looking up the dir via `IModFile.findResource("META-INF/mcdp-mixin-bridges")`.
3. **`registerAutoBridgeManifestsFromIndex(modLoader, indexFile, resolver)` plus an emitted index** (post-v0.1.0 errata). The directory-style `findResource` turned out to behave inconsistently on NeoForge `UnionPath` (returns `null` or a speculative path that throws when listed), so codegen started emitting a `META-INF/mcdp-mixin-bridges-index.txt` file listing every per-mixin manifest filename. Adapters read the index and resolved each `<fqn>.txt` via single-file `findResource`, which IS reliable everywhere.

Each step solved one symptom but left the on-disk shape opinionated: a custom newline-separated triplet format inside multiple per-mixin files, plus a separate index file. A third-party report (`mc-fluid-physics`) flagged that the bespoke triplet parser had zero coverage for ≥2 triplets — when one mixin uses two mod-private targets (e.g. `AbstractBlockStateMixin` referencing `FluidPhysicsConfig` and `FluidPhysicsMod`), readers of the manifest had to handle multiple sequential triplets correctly. Treating the file as a flat `Properties` blob made overlapping `bridge=`/`impl=`/`field=` keys overwrite each other, leaving only the last triplet registered.

The project already ships an in-tree TOML reader (`MiniToml`, ADR-0015) that supports array-of-tables — the natural shape for "one entry per `(mixin, field)` bridge". Migrating drops a custom parser, drops the directory-walking concern entirely, and unifies the on-disk format with the existing `META-INF/mcdepprovider.toml` mod manifest.

## Decision

### File format

One file per mod: `META-INF/mcdp-mixin-bridges.toml`. Replaces both the per-mixin `<fqn>.txt` files and the separate index file.

```toml
# auto-generated by mcdp-mixin-bridges; do not edit
[[bridge]]
mixin = "com.example.mixin.AbstractBlockStateMixin"
field = "LOGIC_FluidPhysicsConfig"
interface = "com.example.mod.mcdp_mixin_bridges.FluidPhysicsConfigBridge"
impl = "com.example.mod.mcdp_mixin_bridges.FluidPhysicsConfigBridgeImpl"

[[bridge]]
mixin = "com.example.mixin.AbstractBlockStateMixin"
field = "LOGIC_FluidPhysicsMod"
interface = "com.example.mod.mcdp_mixin_bridges.FluidPhysicsModBridge"
impl = "com.example.mod.mcdp_mixin_bridges.FluidPhysicsModBridgeImpl"
```

`[[bridge]]` is a TOML array-of-tables. `MiniToml.parse(...)` returns it as `List<Map<String, String>>` — one map per entry, with the four expected keys. Required keys: `mixin`, `field`, `impl`. `interface` is informational (recorded in the manifest for human inspection; the runtime never reads it).

### Codegen

`BridgeCodegenTask` walks every rewritten mixin's target set and emits one `[[bridge]]` table per `(mixinFqn, fieldName)` into the single manifest file. Empty rewrite set → no file emitted (skipped early-return upstream still applies).

### Runtime

`McdpProvider` exposes one entry point:

```java
public static int registerAutoBridgeManifestToml(ModClassLoader modLoader, Path tomlFile)
```

Reads the file via `MiniToml.parse(...)`, iterates the `bridge` array-of-tables, validates the three required keys (`mixin`, `field`, `impl`), registers each entry into the `(mixinFqn#fieldName) → BridgeEntry` map. Missing or null path → returns 0. Malformed TOML or incomplete entry → throws `IllegalStateException`.

The mixin-side LOGIC_* field is still populated lazily by an emitted `<clinit>` via `resolveAutoBridgeImpl(mixinFqn, fieldName)` — that part of the design (carried over from 0018's "lazy clinit wiring" errata) is unchanged.

### Adapter wiring

Both platform adapters now do a single-file lookup and one registration call:

- **NeoForge** (`McdpLanguageLoader.loadMod`): `info.getOwningFile().getFile().findResource("META-INF/mcdp-mixin-bridges.toml")` → call `registerAutoBridgeManifestToml`.
- **Fabric** (`McdpPreLaunch.onPreLaunch`): `mod.findPath("META-INF/mcdp-mixin-bridges.toml").orElse(null)` → call same.

Both adapters log `mcdepprovider: registered N auto-bridge entries for <modId>` regardless of count, so a zero-count outcome surfaces in server logs instead of producing a `<clinit>` crash later.

### `MiniToml` access

`MiniToml` was package-private inside `de.lhns.mcdp.deps`. Promoted to `public` so `core` (which already depends on `:deps-lib` transitively via `api(project(":deps-lib"))`) can call it. Closed-schema caveats stay documented in javadoc.

## Consequences

**Positive**

- Single-file `findResource` lookup works on every loader/mode. NeoForge `UnionPath` directory quirks are moot — we only ask for exact files.
- Eliminates the bespoke triplet parser and its multi-target hazards. The `mc-fluid-physics AbstractBlockStateMixin` regression cannot recur because there is no parser to mis-handle duplicate keys — `MiniToml` returns each `[[bridge]]` table as its own map.
- One on-disk format for both the mod manifest (`mcdepprovider.toml`) and the bridge manifest (`mcdp-mixin-bridges.toml`). Reduces consumer cognitive load and `unzip -p` debug surface.
- No new runtime dependency: `MiniToml` already on `:core`'s classpath via `:deps-lib`.
- Self-validating: each entry is a self-contained table; partial entries fail loudly at parse time, not silently at the next `<clinit>`.

**Negative**

- Existing jars built with the old `.txt` + index format won't load auto-bridges under the new runtime. Mitigation: no released artifacts shipped the old format outside SNAPSHOT — the change is internal to pre-1.0 churn. Any downstream that consumed a SNAPSHOT must rebuild against the same `mcdp` SNAPSHOT.
- `MiniToml`'s public surface grows by one method. The closed-schema caveats remain documented; consumers who need a general-purpose TOML reader still shouldn't use it.

## Alternatives

**Per-mixin `.txt` files with a parser anchored on `field=`.** Rejected: fixes the immediate symptom but keeps the bespoke format. Same engineering cost as migrating; no robustness gain against future schema growth.

**JSON via `MiniJson`.** Equivalent in capability but introduces a second on-disk format (`mcdp-mixin-bridges.json` next to `mcdepprovider.toml`). TOML wins by single-format consistency.

**Encode the bridge map directly into the rewritten mixin's bytecode constant pool.** Eliminates the file lookup entirely. Rejected: opaque to debuggers and `unzip -p`; harder to inspect during third-party diagnosis (the `mc-fluid-physics` report leaned heavily on `unzip -l` and `unzip -p`).

**Walk every `IModFile.getSecureJar().getRootPath()`.** Tightly couples to FML's internal Path types; brittle across NeoForge versions. Single-file lookup avoids the entanglement.

## Errata: filename rename (post-ADR-0021 cleanup)

The manifest filename was renamed `META-INF/mcdp-mixin-bridges.toml` → `META-INF/mcdp-bridges.toml` in the cleanup pass following ADR-0021's generalized seeding. Schema unchanged: still `[[bridge]]` array-of-tables with `mixin`/`field`/`interface`/`impl` keys. The `mixin` key name is historical — for `@EventBusSubscriber`-annotated subscriber classes it holds the subscriber FQN. The runtime can't and doesn't need to distinguish the two cases.

See ADR-0018's "Errata: rename `mcdp_mixin_bridges` → `mcdp_bridges`" for the full set of renamed identifiers.
