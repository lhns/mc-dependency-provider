# ADR-0007 тАФ Dev-mode runs through the production dep-loading path

**Status:** Accepted

## Context

Mod authors iterate using their build tool's in-game launch tasks тАФ `runClient`, `runGameTestServer`, `runServer` on Gradle with ModDevGradle (NeoForge) or Fabric Loom (Fabric). By default these tasks put the mod's `runtimeClasspath` directly on the Minecraft launch classpath. Everything тАФ the mod jar, cats, circe, and so on тАФ is visible to Minecraft via Gradle's normal classloader.

For any other mod-loader ecosystem this is fine. For us it's a problem: **dev mode and production would exercise different code paths.**

- In dev: mod deps are on the Minecraft classpath from the start. The provider's `ManifestConsumer` download path is never exercised, and `ModClassLoader`'s parent-first delegation might resolve `cats.Foo` via the parent (Gradle classpath) rather than the child (SLP cache), shadowing our isolation logic.
- In production: deps come through the provider. If there's a bug in classloader delegation, SHA verification, cache management, or `ModClassLoader` parent-scope rules, it only surfaces after release.

The FluidPhysics cats-integration doc explicitly documents `runClient` failing with `Invalid package name: 'byte'` because the dev classpath bypasses the mod's shipped JPMS workaround. Same class of problem тАФ dev тЙа prod тАФ from a different angle.

## Decision

The Gradle plugin restructures run tasks to load dependencies **via the provider**, exactly as they would in production:

1. **Strip the mod's runtime deps from the Minecraft classpath.** Only Minecraft + NeoForge/Fabric + the provider itself remain. `implementation`, `runtimeOnly`, and other user deps are removed from the run task's classpath configuration.
2. **Pre-warm the provider's cache from the local Gradle cache.** A `prepareSlpDevCache` task runs before each run task; it walks the mod's resolved artifacts and hard-links (or copies, when hard-linking isn't supported) each one into `~/.cache/mc-lib-provider/libs/<sha256>.jar`. Same cache layout the production runtime uses.
3. **Let the provider take it from there.** The mod jar's manifest (generated by `generateModLibraryManifest`) lists the same coords + SHAs the Gradle plugin just pre-warmed. SLP reads the manifest at launch, sees every SHA in the cache, skips download, builds `ModClassLoader`s exactly as in production.

`compileJava` / `compileScala` / `test` are **not** touched. Mod authors continue to declare deps in standard Gradle configurations; compilation and unit tests use Gradle's normal classpath. Only the in-game launch tasks get the restructured classpath.

## Consequences

**Positive:**
- **Dev mode exercises the exact production code path.** Classloader bugs, delegation issues, cache-management problems surface immediately rather than after publishing.
- **Zero network calls in dev.** The Gradle cache already has every artifact; hard-links are O(1) operations that produce bit-identical files. `ManifestConsumer` hits cache on every SHA and skips download.
- **Compile and test ergonomics are untouched.** Mod authors don't need to learn anything new for the normal edit-compile-test loop; standard Gradle behaves normally.
- **SHA-verified dev mode.** If a mod author's Gradle cache has a corrupted artifact (rare), the SHA mismatch surfaces in dev, not in production.

**Negative:**
- **Tighter coupling to run-task internals.** ModDevGradle's `runClient` and Fabric Loom's equivalent expose classpath configuration, but the hook points aren't symmetric. Each needs its own plugin integration code. Upstream changes to either could break us.
- **Hard-linking is filesystem-dependent.** On most Linux/macOS systems it's fast and free. Windows supports it on NTFS but not across drive letters. Falls back to `Files.copy(REPLACE_EXISTING)` when hard-linking fails тАФ same correctness, slightly more disk use.
- **Mod authors who deliberately want to run Minecraft with a different classpath** (e.g. swapping in a patched cats for debugging) lose the Gradle-classpath trick. They can hand-drop the patched jar into `~/.cache/mc-lib-provider/libs/<expected-sha>.jar` as a workaround, but it's less discoverable.

## Alternatives considered

- **Leave run tasks alone; rely on child-first delegation to prefer SLP's cats over the Gradle-classpath cats.** Correct in most cases, but subtly wrong when the SHA on the Gradle classpath differs from the SHA SLP expects. Also means two copies of each dep jar in memory. Worst of all, it fails silently тАФ the provider's code paths aren't fully exercised.
- **Require mod authors to build and test against a real NeoForge install** тАФ rejected. Destroys the fast-iteration loop that makes Gradle run tasks useful.
- **Implement a custom Gradle run task that launches Minecraft with the restructured classpath from scratch** тАФ considered, rejected. Massive scope increase; MDG and Loom already do the heavy lifting. Hooking into their existing tasks is cheaper and means we inherit upstream fixes.
