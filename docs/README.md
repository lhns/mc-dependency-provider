# mc-lib-provider — architecture

A JVM-language mod provider for Fabric and NeoForge (Minecraft 1.21.1+, Java 21). Supports Java, Scala (2.13 + 3), and Kotlin mods, and resolves Maven dependencies per-mod at install time so mods don't need to bundle or shade their deps.

The individual architectural decisions are recorded in [`adr/`](adr/). This document is the high-level synthesis.

## When to choose this loader

- **Stick with the built-in loader** (`javafml` on NeoForge, the default on Fabric) when you need no external Maven deps, or your deps are small enough to JiJ / shadow.
- **Use this loader** when the mod needs external Maven libraries it would rather not bundle. The default situation for Scala mods (cats, circe, cats-effect, fs2, ...) and often the right answer for Java mods using large libraries (e.g. Jackson, Protobuf, Guava forks).

## The two problems this loader solves

### Problem 1 — JPMS rejects Scala ecosystem jars

NeoForge builds JPMS module layers from mod/dep jars via `cpw.mods.securejarhandler`. For a jar without `module-info.class`, an automatic module descriptor is built from the jar's directory entries. `ModuleDescriptor.Builder.packages(...)` rejects Java keywords as package components, and `cats-kernel.jar` ships directories named `cats/kernel/instances/byte/`, `char/`, `short/`, `int/`, `long/`, `float/`, `double/`, `boolean/`. Every attempt to load cats-kernel into a NeoForge module layer fails with `Invalid package name: 'byte' is not a Java identifier`. Existing Scala loaders work around this by maintaining a forked cats with renamed packages, breaking compatibility with libraries compiled against vanilla cats (circe being the obvious casualty).

### Problem 2 — Cross-mod dependency version conflicts

If mod A wants `cats-core:2.10` and mod B wants `cats-core:2.13`, a shared classpath has to pick one version. Maven's standard conflict resolution (highest wins) sometimes works and sometimes doesn't; when it doesn't, users see `NoSuchMethodError` at the worst possible moment. The more mods that rely on external libs, the more often this happens.

## The single mechanism that solves both

**Per-mod classloaders with plain `URLClassLoader` for dependencies, never added to any JPMS module layer.**

- Classes loaded via `URLClassLoader` end up in the loader's **unnamed module**. `ModuleDescriptor.Builder.packages()` is never called for those classes; the keyword-package check never runs. See [ADR-0002](adr/0002-avoid-jpms-via-unnamed-modules.md).
- Each mod gets its own `ModClassLoader`, with the mod jar and the mod's dep jars as URLs. Its parent is the provider's own loader (which sees Minecraft, NeoForge/Fabric, the JDK). Delegation is parent-first for shared platform types (`java.*`, `net.minecraft.*`, `net.neoforged.*`, etc.) and child-first for everything else. Two mods can use two different cats versions without conflict. See [ADR-0001](adr/0001-per-mod-urlclassloaders.md).

One pattern, two wins.

## Architecture

```
mc-lib-provider/
  docs/                             # you are here
    README.md
    adr/                            # architecture decision records
  deps-lib/                         # shared build-tool-agnostic dep library
    Manifest / ManifestIo / ManifestConsumer / ManifestProducer / LibraryCache
  core/                             # loader-agnostic runtime
    ModClassLoader / LoaderCoordinator / EntrypointAdapter + Java/Scala/Kotlin impls
  gradle-plugin/                    # build-time helper for mod authors
    Vendors deps-lib. Thin wrapper that reads Gradle's resolved runtimeClasspath
    and emits a manifest. Future sbt/Maven/CLI siblings follow the same pattern.
  fabric/                           # Fabric LanguageAdapter + PreLaunchEntrypoint
  neoforge/                         # NeoForge IModLanguageLoader
```

Target footprint: provider jar under 300 KB. The heavy lifting (Maven resolution via Aether) is in the Gradle plugin, never in the runtime.

**Build order:** `deps-lib` → `gradle-plugin` (vendors/shades `deps-lib`) → `core` → `fabric` / `neoforge`.

## Dependency manifest

Mods ship `META-INF/mc-jvm-mod.toml`, generated by the Gradle plugin from the resolved `runtimeClasspath`:

```toml
lang = "scala"          # "java" | "scala" | "kotlin"

# Package prefixes whose classes are loaded parent-first by this mod's ModClassLoader.
# Typically the mod's Java-facing API: mixin classes, bridge interfaces, anything
# that must be the same Class object in both NeoForge's game layer and the per-mod
# classloader. See ADR-0008 for why this matters for Mixin.
shared_packages = ["de.lolhens.fluidphysics.api"]

[[libraries]]
coords = "org.scala-lang:scala3-library_3:3.5.2"
url    = "https://repo1.maven.org/maven2/org/scala-lang/scala3-library_3/3.5.2/scala3-library_3-3.5.2.jar"
sha256 = "..."

[[libraries]]
coords = "org.typelevel:cats-core_3:2.13.0"
url    = "..."
sha256 = "..."

# every transitive — scala-library, cats-kernel, cats-effect, circe-*, ...
```

Flat, pre-resolved, SHA-pinned. Transitive resolution happens on the mod author's machine at build time; we freeze the outcome. See [ADR-0003](adr/0003-build-time-dep-resolution.md).

## Boot flow

**NeoForge.** The provider is discovered via `ServiceLoader<IModLanguageLoader>`. For each Scala/Java/Kotlin mod, the provider reads its manifest, downloads missing libs into `~/.cache/mc-lib-provider/libs/<sha256>.jar` (verifying every SHA), then builds a `ModClassLoader` and loads the entrypoint class through it. Event bus, registries, and networking work via reflection on the mod's Class object regardless of which classloader defined it.

**Fabric.** A `PreLaunchEntrypoint` runs the same download/cache flow before any `LanguageAdapter.create(...)` call. `LanguageAdapter.create(...)` loads the mod class through the per-mod `ModClassLoader` and dispatches via the appropriate `EntrypointAdapter`.

## Gradle plugin

Three responsibilities:

1. **Manifest generation.** A task walks Gradle's resolved `runtimeClasspath` and hands coords + repos to `deps-lib`'s `ManifestProducer` (which uses Apache Maven Resolver / Aether). The resulting manifest is written into the mod jar's `META-INF/`.
2. **Dev-mode cache pre-warming.** Before `runClient` / `runGameTestServer`, the plugin hard-links the mod's Gradle-cached artifacts into `~/.cache/mc-lib-provider/libs/<sha>.jar`. No network, cache always warm.
3. **Dev-mode classpath surgery.** The plugin removes the mod's runtime deps from the run task's Minecraft classpath. Only Minecraft + NeoForge/Fabric + the provider remain on the Minecraft classpath; deps are served by the provider's own loader, just as in production.

Normal `compileJava` / `compileScala` / `test` are untouched — mod authors declare deps in standard Gradle configurations. The plugin only touches the in-game run paths and the packaged jar. See [ADR-0007](adr/0007-dev-mode-parity.md).

## Language support

Three `EntrypointAdapter` implementations, picked by the manifest's `lang` field:

- **Java** — plain constructor-arity dispatch: `(IEventBus, ModContainer, Dist)` → `(IEventBus, ModContainer)` → `(IEventBus)` → no-args.
- **Scala** — `MODULE$` lookup for `object` singletons, falling back to the ctor chain for classes.
- **Kotlin** — `INSTANCE` lookup for `object` singletons, falling back to the ctor chain for classes.

The adapter is the only language-specific code. Everything else is shared. See [ADR-0005](adr/0005-pluggable-entrypoint-adapter.md).

## Known limitations

- **Mixin usable via bridge pattern.** Mixin classes must be Java (or avoid direct Scala references) and call into Scala/Kotlin impls via an interface; the impl lives in the per-mod classloader. Shared interfaces go under a manifest-declared `shared_packages` prefix so they're the same `Class` across loaders. See [ADR-0008](adr/0008-mixin-via-bridge-pattern.md).
- **First launch requires network.** The provider downloads per the manifest. A companion `mc-lib-provider-prefetch` CLI (future) pre-populates the cache from a mod set, for modpack authors shipping offline bundles.
- **Scala stdlib duplication.** If N mods pin slightly different stdlib versions (`3.5.2` vs `3.5.3`), each gets its own copy. If they pin the same SHA, the classloader is coalesced (see [ADR-0006](adr/0006-sha-keyed-classloader-coalescing.md)).
- **Cross-mod APIs passing Scala types.** Mod A's `List[String]` and Mod B's `List[String]` are different `Class` objects if each mod has its own Scala stdlib. Cross-mod APIs should use Java-native types or explicit serialization.

## Alternatives considered

See [ADR-0009](adr/0009-reject-alternatives.md) for the full rejection record. Short list: forked cats, per-mod shadowJar, OSGi (Felix/Equinox), Coursier or Aether at runtime, per-mod JPMS module layers, build-time bytecode rewriting.
